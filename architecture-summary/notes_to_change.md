## уточнить четко зачем именно нужен rag. 
в тексте проектирования, было привдеено: "Векторная модель — вспомогательная подсистема для реализации долгосрочной памяти ассистента. Предназначена для хранения эмбеддингов методических указаний, учебных планов и истории взаимодействия, что позволяет языковой модели извлекать семантический контекст при декомпозиции новых задач"

- Нужно четко разграничить назначеине технологи rag. в анализе упомиается что "RAG выступает в роли долгосрочной памяти для статических данных, а LCW применяется в качестве рабочей памяти для оперативного логического анализа." То есть я полагаю что rag больше подходит для flow, где юзер загружает документ -> он погружается в базу -> при запросе к модели на этапе обогащения контекста релеватные части данных извлекюатся через interface vectorstore (spring ai) - на диаграммах этот интерфейс просто внедрён в сервис агента.
И вот я думаю - стоит ли вообще его вырезать -  но тогда работа уже будет менее солидная на мой взгляд, плюс в анализе рассматривается этот подход.
Или же можно дообудить - полагаю тут нужно будет допилить просто на диаграммах энд-поинт под загрузку документа. 
Также для контекста рассуждений можешь использовать appendix.tex - там пример системного промпта.
