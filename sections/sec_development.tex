\section{Разработка персонального ассистента организации времени обучающихся} 
\label{sec:development}

В данном разделе приведены обоснование используемых средств разработки, а также демонстрация её результатов. Для доказательства работоспособности системы и её соответствия требованиям был также разработан тестовый пример.

\subsection{Обоснование технических и программных средств разрабатываемой системы}

Выбор программных средств определяется архитектурными решениями, зафиксированными в разделе проектирования, а именно: модульным монолитом с трёхуровневым разделением ответственности, транзакционной изоляцией черновиков расписания и агентным модулем, совмещающим вызов языковой модели с детерминированным алгоритмом планирования. Каждое средство разработки соответствует конкретному слою системы и обосновывается техническими требованиями к нему.

\subsubsection{Серверная часть}

Серверная часть системы реализована на языке программирования Java версии 21, являющемся актуальным выпуском с долгосрочной поддержкой. Строгая статическая типизация языка обеспечивает надёжность доменной модели на этапе компиляции и снижает вероятность ошибок при приведении ответов языковой модели, поступающих в формате JSON, к объектам передачи данных. Зрелость экосистемы Java и наличие специализированных библиотек для интеграции с языковыми моделями также является определяющим фактором выбора.

В качестве основного каркаса приложения применяется Spring Boot версии 3.4. Данная платформа предоставляет механизмы автоконфигурации, встроенный веб-сервер и производственно-готовую инфраструктуру для построения REST-приложений без необходимости в ручной настройке большинства компонентов. Модуль Spring Web реализует слой контроллеров, обеспечивая маршрутизацию HTTP-запросов, десериализацию входящих данных и первичную валидацию полей объектов передачи данных.

Для разграничения доступа к ресурсам и управления сессиями используется модуль Spring Security совместно с механизмом токенов в формате JWT. Клиент передаёт токен в заголовке каждого запроса, а фильтр безопасности проверяет его подпись и срок действия до передачи управления контроллеру. Такой подход соответствует требованию безопасности REST-архитектуры, поскольку состояние сессии не хранится на сервере.

\subsubsection{Система хранения данных}

В качестве реляционной системы управления базами данных выбрана PostgreSQL версии 17. Ключевым аргументом в её пользу является нативная поддержка ограничений исключения, позволяющих объявить непосредственно на уровне схемы правило, запрещающее сохранение перекрывающихся временных диапазонов для зафиксированных слотов одного пользователя. Данный механизм обеспечивает физическую защиту от коллизий расписания без дублирования логики проверки в прикладном коде. Помимо этого, PostgreSQL предоставляет полноценную поддержку транзакций с настраиваемыми уровнями изоляции, необходимую для атомарного применения черновика расписания к основной таблице слотов.

Взаимодействие серверного приложения с базой данных организовано через модуль Spring Data JPA, реализующий паттерн «Репозиторий» посредством объектно-реляционного отображения. Применение данного модуля позволяет бизнес-логике оперировать исключительно объектами доменной модели, оставаясь независимой от конкретных запросов к базе данных, что соответствует архитектурному принципу слабой связности слоёв системы.

Управление версиями схемы базы данных производится инструментом Liquibase. При каждом запуске приложения инструмент последовательно применяет сценарии миграции, не применявшиеся ранее, что обеспечивает воспроизводимость структуры базы данных в любой среде развёртывания и исключает рассинхронизацию схемы с кодом приложения.

\subsubsection{Агентный модуль}

Интеграция с внешним программным интерфейсом языковой модели реализована посредством библиотеки Spring AI версии 1.0. В качестве конкретного поставщика языковой модели используется сервис OpenRouter, предоставляющий унифицированный программный интерфейс к широкому спектру языковых моделей различных производителей. Совместимость OpenRouter с протоколом OpenAI API позволяет подключить его к Spring AI без изменений кода сервисного слоя --- достаточно переопределить базовый адрес запросов и ключ авторизации в конфигурационном файле приложения. Обязательным требованием к выбираемой модели является поддержка режима структурированного вывода и механизма вызова функций, поскольку оба используются в реализации агентного модуля.

Режим структурированного вывода принудительно ограничивает генерацию модели заданной схемой данных, гарантируя синтаксическую корректность ответа. В Spring AI данный механизм реализован через класс \texttt{BeanOutputConverter}, который транслирует описание Java-класса объекта передачи данных в схему и валидирует ответ модели до десериализации. Это исключает получение некорректных ответов, которые языковая модель могла бы сгенерировать в произвольном формате.

Реализация инструментов агентного модуля, то есть функций, вызываемых языковой моделью в ходе обработки запроса, обеспечивается механизмом вызова функций Spring AI. Методы Java, аннотированные \texttt{@Tool}, регистрируются в контексте сеанса автоматически, а языковая модель принимает решение об их вызове самостоятельно на основании содержания запроса пользователя. Именно через данный механизм декомпозированные параметры задачи передаются из языковой модели в детерминированный алгоритм планирования.

\subsubsection{Клиентская часть}

Пользовательский интерфейс реализован в виде одностраничного приложения на основе библиотеки React версии 18. Компонентная модель React обеспечивает декларативное управление состоянием интерфейса и позволяет реализовать интерактивные карточки черновика расписания как самостоятельные переиспользуемые компоненты, обновляющиеся при получении данных от сервера.

Для отображения календарной сетки применяется библиотека FullCalendar в редакции для React. Данная библиотека предоставляет готовый компонент с поддержкой представлений по неделям и дням, программного управления событиями и их перетаскивания в пределах сетки. Возможность перетаскивания событий непосредственно обеспечивает описанный в разделе проектирования сценарий ручного редактирования черновика расписания без дополнительного взаимодействия с агентом. Использование данной библиотеки исключает необходимость самостоятельной разработки логики позиционирования и отрисовки временных блоков.

Взаимодействие клиентского приложения с сервером организовано через библиотеку Axios. Перехватчики запросов данной библиотеки обеспечивают автоматическое добавление JWT-токена в заголовок каждого исходящего запроса, устраняя необходимость явного управления авторизацией в каждом компоненте интерфейса.

\subsubsection{Инфраструктура}

Для обеспечения воспроизводимости среды разработки и упрощения локального развёртывания применяется платформа контейнеризации Docker совместно с инструментом Docker Compose. Конфигурационный файл Docker Compose описывает два сервиса: контейнер с серверным приложением и контейнер СУБД PostgreSQL. Такая конфигурация позволяет развернуть полностью работоспособное окружение без ручной установки зависимостей и настройки базы данных, что особенно важно для воспроизведения тестового примера при демонстрации работоспособности системы.


\subsection{Обзор результатов разработки}

В рамках разработки реализованы серверное приложение на платформе Spring Boot, включающее слои контроллеров, бизнес-логики и доступа к данным, детерминированный алгоритм планирования временных слотов, механизм агентного взаимодействия с языковой моделью и клиентское одностраничное приложение с гибридным интерфейсом. В данном подразделе приведено описание ключевых нетривиальных решений, реализованных в ходе разработки.

\subsubsection{Алгоритм планирования временных слотов}

На вход алгоритм принимает упорядоченный список подзадач и объект состояния календаря, содержащий перечень зафиксированных временных слотов в диапазоне от текущего момента до дедлайна задачи. Упорядочивание подзадач выполняется предварительно посредством топологической сортировки по полю \texttt{dependency\_id}, что гарантирует расстановку логически зависимых этапов в правильной последовательности.

Псевдокод алгоритма приведён в листинге~\ref{lst:algorithm}.

\begin{lstlisting}[basicstyle=\small\ttfamily, frame=single,
    caption={Pseudocode of the planning algorithm},
    label={lst:algorithm}]
Input: subtasks -- list of subtasks (topologically sorted by dependency_id)
      calendarState -- {busySlots, deadline, workingHours}
Output: scheduledSlots, unschedulable

freeWindows <- BuildFreeWindows(calendarState)
scheduledSlots <- []
unschedulable  <- []

for each subtask s in subtasks:
  placed <- false
  for each window w in freeWindows:
    if (w.end - w.start) >= s.estimatedMinutes:
      append Timeslot(w.start, w.start + s.estimatedMinutes) to scheduledSlots
      w.start <- w.start + s.estimatedMinutes + MIN_GAP
      placed <- true
      break
  if NOT placed:
    append s to unschedulable

return (scheduledSlots, unschedulable)

BuildFreeWindows(calendarState):
  windows <- []
  for each working day d in [now, deadline]:
    dayInterval <- [d + workStart, d + workEnd]
    windows <- windows + Subtract(dayInterval, busySlots(d))
  return windows, sorted by ascending start
\end{lstlisting}

Выбор осуществляется без возврата: ни одно ранее принятое решение не пересматривается, что обеспечивает линейную вычислительную сложность относительно числа подзадач. Если свободного времени до дедлайна оказалось недостаточно, незапланированные подзадачи возвращаются в составе черновика вместе с соответствующим уведомлением.

\subsubsection{Реализация агентного модуля}

Агентный модуль реализован в компоненте \texttt{AgentService}. Инструменты агента объявлены как методы Java, аннотированные \texttt{@Tool}, с декларативным текстовым описанием их назначения. На основе этого описания языковая модель самостоятельно определяет, какой инструмент необходимо вызвать и с какими параметрами, не требуя явной логики ветвления в прикладном коде.

Для получения структурированного результата декомпозиции применяется \texttt{BeanOutputConverter}, которому передаётся Java-класс \texttt{DecomposedTaskDTO}. Конвертер автоматически формирует JSON-схему из полей класса и встраивает её в системный промпт, обязывая языковую модель сформировать ответ строго в соответствии со схемой. Это исключает необходимость ручного разбора текстового ответа и делает интеграцию устойчивой к вариациям в формулировках модели.

Пример объявления инструмента агента, принимающего параметры декомпозированной задачи от языковой модели, приведён в листинге~\ref{lst:tool}.

\begin{lstlisting}[language=Java, basicstyle=\small\ttfamily, frame=single,
    caption={Пример объявления инструмента агентного модуля},
    label={lst:tool}]
@Component
public class SchedulingTools {

    private final ScheduleService scheduleService;

    @Tool(description = "Decompose a learning task and schedule it. "
                      + "Extract title, deadline and subtasks from user request.")
    public DraftResponseDTO scheduleTask(
            String title,
            String deadlineIso,
            List<SubtaskParam> subtasks) {
        // title, deadlineIso, and subtasks are extracted by the model
        // from the user request text automatically
        ScheduleRequestDTO request =
            new ScheduleRequestDTO(title, deadlineIso, subtasks);
        return scheduleService.processDecomposition(request);
    }
}
\end{lstlisting}

\subsubsection{Жизненный цикл черновика расписания}

Черновик расписания реализует конечный автомат с тремя состояниями: \texttt{PENDING}, \texttt{APPROVED} и \texttt{DECLINED}. Управление переходами сосредоточено в \texttt{ScheduleService}.

При создании черновика сервис сохраняет объекты задачи и подзадач, устанавливая статус задачи \texttt{IN\_DRAFT}, после чего вызывает алгоритм планирования. Полученные временные слоты сохраняются с флагом \texttt{is\_committed = false} и связываются с записью черновика в статусе \texttt{PENDING}.

При подтверждении черновика сервис выполняет единственную атомарную транзакцию: устанавливает \texttt{is\_committed = true} для всех слотов черновика, переводит статус черновика в \texttt{APPROVED} и статус задачи в \texttt{SCHEDULED}. Ограничение исключения базы данных на данном шаге проверяет отсутствие временных коллизий и отклоняет транзакцию в случае их обнаружения.

При отклонении черновика временные слоты удаляются, статус черновика устанавливается в \texttt{DECLINED}, а статус задачи возвращается в \texttt{UNSCHEDULED}. Объекты задачи и подзадач сохраняются в базе данных, переходя в состояние бэклога и оставаясь доступными для повторного планирования.

\subsection{Вывод}

В результате разработки реализован программный комплекс персонального ассистента организации времени обучающихся, соответствующий архитектурным решениям и функциональным требованиям, зафиксированным в разделе проектирования.

Серверная часть системы построена в виде модульного монолита на платформе Spring Boot с трёхуровневым разделением ответственности. Слой контроллеров обеспечивает приём и первичную валидацию запросов; сервисный слой реализует бизнес-логику, включая управление жизненным циклом черновика и оркестрацию вызовов агентного модуля; слой доступа к данным инкапсулирует взаимодействие с СУБД через интерфейсы репозиториев.

Нейросимволический подход к планированию реализован посредством чёткого разделения ответственности между двумя компонентами. Языковая модель, интегрированная через Spring AI с поставщиком OpenRouter, выполняет декомпозицию задачи и возвращает структурированный объект передачи данных, схема которого гарантируется механизмом \texttt{BeanOutputConverter}. Жадный алгоритм компонента \texttt{ScheduleAlgorithm} принимает декомпозированные подзадачи и распределяет их по свободным временным окнам с учётом зависимостей между этапами и рабочего диапазона дня.

Транзакционная целостность данных обеспечена на двух уровнях: на уровне приложения — за счёт изоляции черновика как конечного автомата с явными переходами между состояниями \texttt{PENDING}, \texttt{APPROVED} и \texttt{DECLINED}; на уровне базы данных — за счёт ограничений исключения PostgreSQL, физически исключающих пересечение зафиксированных временных диапазонов при подтверждении черновика.

Клиентское одностраничное приложение реализует гибридный агентный интерфейс: диалоговое окно позволяет формулировать намерения в произвольной текстовой форме, а интерактивная календарная сетка на основе FullCalendar обеспечивает визуальное редактирование и подтверждение сгенерированного черновика расписания. Воспроизводимость среды развёртывания обеспечена средствами Docker Compose.
